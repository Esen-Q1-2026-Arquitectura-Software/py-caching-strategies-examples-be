What is Response Caching Middleware?

A Starlette middleware (BaseHTTPMiddleware) wraps the entire FastAPI request/response pipeline. Before a GET request reaches any route handler, the middleware checks Redis for a cached response. On a HIT it returns immediately — the route handler body never runs. On a MISS it calls the handler, stores the full JSON response in Redis, and returns it with an X-Cache: MISS header. Every subsequent request gets X-Cache: HIT until TTL expires.

Key benefit
All GET routes in the namespace get caching for free — zero lines of cache code in any route handler.
HIT path
Middleware reads Redis, reconstructs HTTP response. Route handler never executes. < 2 ms.
MISS path
Middleware calls handler (DB query ~150 ms), streams body into Redis, returns with X-Cache: MISS.

Middleware Request Flow

TTL: 30 s — Key prefix: mc: — Scope: /v1/middleware/*

GET /v1/middleware/catalog (route handler — zero Redis commands)
1. ResponseCacheMiddleware.dispatch(request) ← runs before any route
2. cache_key = "mc:/v1/middleware/catalog"
3a. HIT: redis.GET mc:/v1/middleware/catalog → Response(X-Cache: HIT) < 2 ms — handler skipped
3b. MISS: call_next(request) → route handler runs (~150 ms DB)
3c. redis.SETEX mc:... 30 <response body> ← stored by middleware
4. return Response(X-Cache: MISS)

Middleware vs Cache-Aside

Same caching outcome — completely different ownership of cache logic

2.7 Response Middleware (this demo)
# Middleware handles ALL routes automatically
class ResponseCacheMiddleware(BaseHTTPMiddleware):
  async def dispatch(req, call_next):
    key = f"mc:{req.url.path}"
    cached = await redis.get(key)
    if cached:
      return Response(X-Cache="HIT")
    resp = await call_next(req)
    await redis.setex(key, 30, body)
    return Response(X-Cache="MISS")

# Route handler — ZERO cache logic
@app.get("/v1/middleware/catalog")
async def catalog(db):
    tracks = await db.execute(select(Track))
    return {"tracks": tracks}
2.3 Cache-Aside (manual)
# Endpoint owns ALL cache logic


@app.get("/v1/cache-aside/order/{id}")
async def endpoint(order_id, redis, db):

  # 1. Check cache manually
  raw = await redis.get(f"order:{order_id}")
  if raw:
    return json.loads(raw)

  # 2. DB fetch on miss
  data = await db.get(Order, order_id)

  # 3. Populate cache manually
  await redis.setex(key, 300, json.dumps(data))
  return data

Middleware vs Per-Route Caching Patterns

Aspect 2.7 Middleware 2.3 Cache-Aside 2.6 Read-Through
Cache logic location Middleware class — zero app code Inside each endpoint @cached decorator on DB function
Granularity Full HTTP response (coarse-grained) Per value, custom key Per function call argument
HIT latency < 2 ms (route never called) < 2 ms (redis.get) < 2 ms (aiocache)
TTL Fixed per middleware instance (30 s demo) Configurable per call Fixed at decorator definition
Response header insight X-Cache: HIT / MISS on every response Only if you add it manually Only if you add it manually
Bypass per-request Yes — via X-No-Cache header pattern Yes — skip the check in code No — decorator always applies
Best for Public GET APIs, zero-code caching, CDN-like behaviour Selective caching, complex TTL logic Read-heavy functions, clean decorators
