How this demo works

Cache MISS
Simulates a slow database read (120–200 ms). The result is then stored in the cache. Next call will instantly be a HIT.
Cache HIT
Data returned directly from in-process RAM or Redis in < 2 ms. The database is never touched.
Clear Cache
Evicts all entries. The subsequent fetch will be a MISS again — so you can observe the full cycle.

2.1 — In-Memory Caching

Data cached directly in Python process RAM. Nanosecond access, zero network overhead. Lost on restart. Not shared across multiple workers.

functools.lru_cache No TTL
maxsize=128

Caches country config forever (until eviction). No expiry — right for static reference data.

cachetools.TTLCache TTL: 30s
maxsize=100

Caches user permissions for 30 s, then auto-expires. Thread-safe via threading.Lock().

2.2 — Redis — The Core Backend Cache

Distributed cache. Survives server restarts and is shared across all FastAPI workers / instances. Uses SETEX for atomic set+TTL and GET for retrieval. Connection pool initialized in lifespan().

Redis SETEX / GET (cache-aside) TTL: 300s
max_connections=20

Cache key product:{id}. Pattern: check Redis → miss → DB → SETEX → return.

GET product:{id} → HIT: return JSON ~1ms
→ MISS: DB fetch ~200ms → SETEX product:{id} 300 <json> → return
Why Redis over in-memory?
+ Shared across all workers / pods
+ Survives app restart
+ Atomic TTL via SETEX
+ Rich data types (strings, hashes, sets)
- Network round-trip (~0.5–2ms)
- External dependency

2.3 — Cache-Aside (Lazy Loading) Pattern

Uses a real SQLAlchemy ORM query on a cache miss — not a fake in-memory dict. Adds negative caching: when an ID does not exist the null result is stored with a short 15 s TTL so repeated 404 lookups never reach the database.

Redis + SQLAlchemy (Cache-Aside) positive TTL: 120s — negative TTL: 15s
orders table → SQLite

Cache key order:{id}. On miss: real SELECT via SQLAlchemy ORM. Null results also cached (15 s) to prevent DB hammering on repeated 404s.

GET order:{id}
→ HIT (positive): return cached order   ~1ms — DB not touched
→ HIT (negative): return 404 from nil sentinel   ~1ms — DB NOT touched
→ MISS: SQLAlchemy SELECT   ~180ms → found → SETEX order:{id} 120 <json>   /   not found → SETEX order:{id} 15 "null"
Key upgrades vs 2.2 Redis
+ Real SQLAlchemy ORM query on miss (not fake dict)
+ Negative caching — null cached 15s
+ Lazy population — only accessed IDs are cached
+ Two separate TTLs: 120s positive, 15s negative
! Try ID 99 twice — second call skips DB entirely

2.4 — Write-Through Pattern

Every PUT writes to SQLite DB and Redis simultaneously before responding. The cache is always consistent with the database — no stale reads possible after any write. Contrast with Cache-Aside: a bare DB write there would leave stale data in Redis until TTL expires.

Write-Through — Pattern Diagram

DB write and cache write happen in the same request before the response is sent.

PUT /profile/{id} <body>
→ 1. UPDATE user_profiles SET … WHERE id={id}   (SQLAlchemy / SQLite)
→ 2. SETEX profile:{id} 600 <updated json>   (Redis — cache updated atomically)
→ 3. return updated profile
→ subsequent GET profile:{id} is always a cache HIT
Read Profile TTL: 600s

After any PUT, this GET is always a cache HIT — the cache is kept current by every write.

Update Profile (Write-Through)

Writes DB + Redis atomically. The next GET is guaranteed to return this exact data.

+ Cache always consistent — zero stale reads
+ Next GET is a guaranteed cache HIT
- Write latency is slightly higher (2 writes per PUT)
! Update a profile, then GET it — always a HIT

Strategy Comparison

Strategy TTL Scope Thread-safe Survives restart Best for
lru_cache None Process Yes (GIL) No Immutable config, small data sets
TTLCache Configurable Process Yes (Lock) No Semi-static data (permissions, sessions)
Redis SETEX/GET Configurable Distributed Yes (atomic) Yes Production, multi-instance, large payloads
Cache-Aside 120s / 15s neg Distributed Yes (atomic) Yes Read-heavy, ORM-backed, negative caching
Write-Through 600s Distributed Yes (atomic) Yes Frequently updated data, zero stale tolerance
