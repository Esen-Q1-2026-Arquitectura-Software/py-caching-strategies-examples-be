Cache Stampede Prevention

When a popular key expires, a burst of concurrent requests all see a cache miss and simultaneously query the database — the stampede effect. The fix: a distributed Redis NX lock ensures only one request rebuilds the cache while all others spin-wait and then read the freshly populated value.

Core pattern
cached = await redis.get(key)
if cached:  return json.loads(cached)              # fast path

acquired = await redis.set(lock_key, "1", nx=True, ex=10)   # atomic

if acquired:
    try:
        result = await expensive_db_call()         # only ONE request runs this
        await redis.setex(key, ttl, json.dumps(result))
        return result
    finally:
        await redis.delete(lock_key)               # always release
else:
    for _ in range(20):                            # spin-wait up to 2 s
        await asyncio.sleep(0.1)
        cached = await redis.get(key)
        if cached:  return json.loads(cached)      # lock holder wrote it
    return await expensive_db_call()               # timeout fallback

How the lock prevents stampede

① All N concurrent requests check the cache — all see a MISS (key expired or cold).
② All N race to acquire the lock with SET lock_key "1" NX EX 10. Only one wins (atomic Redis operation).
③ The winner runs the expensive DB call (150 ms), populates the cache, then deletes the lock.
④ The N-1 waiters spin-wait (asyncio.sleep(0.1)) and retry the cache every 100 ms.
⑤ Waiters find the cache populated ≈ 150 ms later — 1 DB call total vs N without protection.
