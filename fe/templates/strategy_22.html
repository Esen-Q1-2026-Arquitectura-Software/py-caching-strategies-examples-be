What is Redis Caching?

Redis is an in-memory data store that runs as a separate process (or cluster). Unlike in-process caches, it is shared across all workers and application instances, survives app restarts, and supports rich atomic operations. The standard pattern for FastAPI: initialise a connection pool once in lifespan(), then GET on every read and SETEX (set + TTL) on every cache-population event.

Scope
All workers, all instances — fully shared.
Survival
Persists across app restarts (until TTL or eviction).
TTL
Set atomically with SETEX. Entries expire automatically.
Best for
Product catalogues, API responses, session tokens, computed results.

Cache-Aside Read Pattern (SETEX / GET)

Key: product:{id} — TTL: 300 s — Connection pool: max_connections=20

GET product:{id}
✓ HIT → return JSON blob ~1 ms — DB not touched
✗
MISS → SELECT from DB ~200 ms
→ SETEX product:{id} 300 <json> one call, atomic set+TTL
→ return data

💡 Fetch a product twice. First call: ~200 ms (DB). Second call: <2 ms (Redis). Click Clear to evict all keys and repeat the cycle.

Code walkthrough
cached = await redis.get(key)
if cached:
return json.loads(cached) # HIT
# miss path
row = await db.get(Product, id)
await redis.setex(key, 300, json.dumps(row))
return row # MISS → now cached
