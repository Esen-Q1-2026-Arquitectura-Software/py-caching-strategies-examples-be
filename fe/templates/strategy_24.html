What is Write-Through Caching?

On every write (PUT), the application updates both the database and the Redis cache synchronously in the same request before responding. This eliminates stale reads: any GET that follows a PUT will always hit a fresh cache value. The trade-off is slightly higher write latency (two writes instead of one), but reads are always fast and always consistent.

On Write (PUT)
UPDATE in SQLite and SETEX in Redis before ack.
On Read (GET)
Almost always a cache HIT since every PUT warms the cache.
TTL
600 s (10 min). Refreshed on every write.
Best for
User profiles, settings — data written often but must be consistent.

Write-Through Flow

Key: profile:{user_id} — TTL: 600 s

PUT /profile/{id} (write path)
1. UPDATE user_profiles SET name=?, bio=? WHERE id=? ← SQLAlchemy
2. SETEX profile:{id} 600 <updated_json> ← Redis (same request)
3. return 200 OK ← both writes confirmed
GET /profile/{id} (read path)
1. GET profile:{id} → HIT (cache always current after any PUT)
MISS only on cold start / after cache clear → DB fallback warms cache once.

Read Profile (GET)

Fetch profile from cache. First call may miss; after any PUT it will always HIT.

Write-Through vs Cache-Aside

Aspect Write-Through (2.4) Cache-Aside (2.3)
Write flow PUT → DB + Cache (atomic) PUT → DB only (cache stale until TTL)
Read after write Always cache HIT Cache MISS (stale until re-populated)
Consistency Strong (no stale reads) Eventual (TTL window)
Write cost Higher (2 writes per PUT) Lower (1 write per PUT)
Best for User profiles, config, settings Read-heavy, rare writes
