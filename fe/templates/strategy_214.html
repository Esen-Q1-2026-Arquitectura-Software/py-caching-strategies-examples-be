Negative Caching & Bloom Filters

Classic DDoS vector: flood your API with non-existent IDs — each request misses the cache and hits the DB. Two layered defences prevent this: a Negative Cache stores "this ID doesn't exist" in Redis, and a Bloom Filter intercepts requests before even hitting Redis, using ~1.2 MB to represent 1 million IDs.

    return await redis.execute_command(

How Bloom Filters work

A Bloom filter uses k independent hash functions to set k bits in a bit array. To check: hash the element, check all k bits. If any bit is 0 → definitely absent. If all k bits are 1 → probably present (but might be false positive).

The Redis BF.RESERVE command lets you specify capacity (expected items) and error rate. At 1% FP rate, 1 million items ≈ 1.2 MB. At 0.1% FP rate ≈ 1.8 MB. This is orders of magnitude less than storing the IDs themselves. 

BF.RESERVE users:bloom 0.01 1000000
# error_rate=1%, capacity=1M → ~1.2 MB

BF.ADD    users:bloom "42"    # insert
BF.EXISTS users:bloom "42"    # → 1 (might exist)
BF.EXISTS users:bloom "99999" # → 0 (definitely absent)

Bloom filter cannot delete. When a user is deleted, the BF still says "maybe exists." The negative cache sentinel handles this: it returns NOT_FOUND within 30 s, without hitting the DB.

False positives are benign here. A false positive means a non-existent ID passes the BF check — then a Redis lookup returns the negative sentinel (or a DB miss which populates it). Maximum cost: one Redis GET. Still far cheaper than a full DB query.

Layer the defences. BF → negative cache → positive cache → DB. Each layer is orders of magnitude cheaper than the next. Even a 1% BF false-positive rate means 99% of non-existent IDs never reach Redis.
