What is Write-Behind Caching?

Write to the cache immediately and acknowledge the client in < 1 ms. The actual database write happens asynchronously in the background — a worker drains the Redis Stream queue every 5 seconds and bulk-inserts into SQLite. This is the fastest possible write acknowledgement at the cost of a brief data-loss window if both Redis and the app crash simultaneously before a flush.

Write ack
Client gets response in < 1 ms — before DB write.
Queue
Redis Stream holds pending events until flushed.
Flush interval
Background worker runs every 5 s, batch-inserts all pending.
Risk
Data in stream but not yet DB can be lost on crash.

Write-Behind Flow (Redis Stream)

Stream key: wb:events — Flush every 5 s — Batch size: 50

POST /event (client request — completes in < 1 ms)
1. XADD wb:events * event_type click user_id 3 ... ← Redis Stream
2. return 202 Accepted ← DB NOT yet written
Background Worker (runs every 5 s)
3. XRANGE wb:events - + ← drain all pending
4. INSERT INTO event_logs VALUES ... (batch) ← SQLAlchemy bulk
5. DEL wb:events ← stream cleared

Submit Event (POST) — Write-Behind

Write-Behind vs Write-Through

Aspect Write-Behind (2.5) Write-Through (2.4)
Write ack < 1 ms (queued, not persisted) ~5–20 ms (DB + cache written)
DB write timing Deferred (batch, every 5 s) Synchronous (in same request)
DB load Low (bulk batch inserts) 1 write per request
Data safety Risk: unflushed data lost on crash Safe: DB written synchronously
Best for High-throughput events, analytics, audit logs User data, settings — consistency is required
