Two-Level Cache (L1 + L2)

L1 is a process-local TTLCache (cachetools) — pure dict lookup, no network, nanosecond latency. L2 is Redis — shared across all instances, ~1 ms. L3 is the database — simulated 100 ms. Each level backfills the one above it.

Core pattern
if key in _l1:                        # L1 check — ns latency
    return _l1[key]                   # HIT — no I/O

raw = await redis.get(l2_key)         # L2 check — ~1 ms
if raw:
    result = json.loads(raw)
    _l1[key] = result                 # backfill L1
    return result

result = await db.fetch(key)          # L3 — ~100 ms
await redis.setex(l2_key, 60, ...)    # populate L2
_l1[key] = result                     # populate L1
return result

Write-Through Demo

A write must update both L1 and L2 to prevent stale reads. Here we:

  1. Persist the new value to the simulated DB
  2. Write to L2 (Redis) with fresh TTL
  3. Evict the L1 entry — next read will backfill from L2
This avoids a window where L1 holds a stale value while L2 is already updated.

Why evict L1 on write?
Without eviction (WRONG)
write DB ✓
write L2 (Redis) ✓
L1 still: old value ← stale!

Next read (within 5 s):
  L1 hit → returns OLD value
  User sees stale data for up to 5 s
With eviction (CORRECT)
write DB ✓
write L2 (Redis) ✓
del _l1[key]     ← evict stale L1

Next read:
  L1 miss → L2 hit → new value
  Backfill L1 with NEW value ✓

L1 is process-local — in a multi-instance deployment (Kubernetes, multiple workers) each pod has its own independent L1. Stale window = L1 TTL (5 s). This is acceptable for most config/reference data. For financial prices or inventory counts, set L1 TTL to 0 or skip L1 entirely. L2 (Redis) is the shared source of truth — invalidate it to propagate changes to all instances within one L2 TTL cycle.
