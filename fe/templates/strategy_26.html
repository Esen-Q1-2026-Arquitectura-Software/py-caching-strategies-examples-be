What is Read-Through Caching?

The cache layer itself is responsible for fetching from the database on a miss. Application code only talks to the cache — it never issues GET / SETEX calls directly. Here, aiocache's @cached decorator intercepts every function call, checks Redis, and only executes the function body (the DB query) when there is a miss — then stores the result automatically.

HIT path
aiocache returns from Redis in < 2 ms. The decorated function body never runs.
MISS path
aiocache calls the function body (DB query ~180 ms), stores result in Redis, returns it.
Key distinction
No redis.get() or redis.setex() anywhere in endpoint or service code.

Read-Through Flow

TTL: 240 s — Namespace: rt — Key pattern: rt:article:{id}

GET /v1/read-through/article/{id} (endpoint code — zero Redis commands)
1. data = await _rt_fetch_article(article_id) ← only line in endpoint touching data
2a. HIT: aiocache.GET rt:article:{id} → returns cached dict (< 2 ms)
2b. MISS: aiocache calls function body → DB query (~180 ms)
2c. aiocache.SET rt:article:{id} EX 240 ← stored automatically
3. return {data, cache_hit, latency_ms, ...}

Read-Through vs Cache-Aside

Same outcome — different ownership of cache logic

2.6 Read-Through (this demo)
# The @cached decorator IS the entire cache layer
@cached(ttl=240, cache=Cache.REDIS,
        namespace="rt",
        serializer=JsonSerializer(),
        key_builder=_rt_key_builder)
async def _rt_fetch_article(article_id):
    # executes ONLY on cache miss
    return await db.get(CatalogItem, article_id)

# Endpoint — zero Redis commands
@app.get("/v1/read-through/article/{id}")
async def endpoint(article_id):
    data = await _rt_fetch_article(article_id)
    return {"data": data}
2.3 Cache-Aside (manual)
# Endpoint owns ALL cache logic

@app.get("/v1/cache-aside/order/{id}")
async def endpoint(order_id, redis, db):

    # 1. Check cache manually
    raw = await redis.get(f"order:{order_id}")
    if raw:
        return json.loads(raw)

    # 2. DB fetch on miss
    data = await db.get(Order, order_id)

    # 3. Populate cache manually
    await redis.setex(key, 300, json.dumps(data))
    return data

Read-Through vs Cache-Aside

Aspect Read-Through (2.6) Cache-Aside (2.3)
Cache logic location Inside @cached decorator — zero app code Inside the endpoint — GET → miss → DB → SETEX
HIT latency < 2 ms (Redis via aiocache) < 2 ms (redis.get)
MISS latency ~180 ms (DB fetch + auto-store) ~180 ms (DB fetch + manual SETEX)
Requires cache library Yes — aiocache (or similar) No — plain redis-py
Code cleanliness Endpoint is pure business logic Endpoint mixes cache + business logic
Custom TTL per call Fixed at decorator definition Flexible — set per request
Best for Read-heavy, uniform data, DRY codebase Selective caching, complex logic, negative TTLs
