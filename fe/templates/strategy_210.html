Database Query Caching

Cache arbitrary SQL query results in Redis using a deterministic hash of the query name + bound parameters as the key. Identical queries with identical parameters hit Redis in < 1 ms; any parameter change produces a distinct key and hits the DB.

Core pattern
key = "dq:" + md5(f"{query_name}:{json.dumps(params, sort_keys=True)}").hexdigest()

cached = await redis.get(key)
if cached:
    return json.loads(cached)            # HIT — Redis < 1 ms

async with session:                      # MISS — real SQLAlchemy query
    result = await session.execute(text(sql), params)
    rows = [dict(r._mapping) for r in result.fetchall()]

await redis.setex(key, ttl, json.dumps(rows))

Filtered Employee List

The query filters employees by department, sort order, hire year, and active status. Each unique combination of parameters produces a distinct cache key. Two requests with the same filters share the same Redis key and the second call is always a HIT.

Request history
SQL executed on MISS
SELECT id, name, department, role, salary, hire_year, active
FROM employees
WHERE (:department IS NULL OR department = :department)
  AND (:hire_year_min IS NULL OR hire_year >= :hire_year_min)
  AND (:active_only = 0 OR active = 1)
ORDER BY
  CASE WHEN :sort_col = 'salary_desc'    THEN salary    END DESC,
  CASE WHEN :sort_col = 'salary_asc'     THEN salary    END ASC,
  CASE WHEN :sort_col = 'hire_year_desc' THEN hire_year END DESC,
  CASE WHEN :sort_col = 'hire_year_asc'  THEN hire_year END ASC,
  id ASC

Department Aggregation

Full-table GROUP BY with 5 aggregate functions — the most expensive query in this demo. No parameters → single stable cache key → effectively 100% hit rate after the first call. First call is ~80 ms; all subsequent calls under TTL are < 1 ms (~80–100× speedup).

Request history
SQL executed on MISS
SELECT
  department,
  COUNT(*)              AS headcount,
  ROUND(AVG(salary), 2) AS avg_salary,
  MAX(salary)           AS max_salary,
  MIN(salary)           AS min_salary,
  SUM(salary)           AS total_payroll
FROM employees
WHERE active = 1
GROUP BY department
ORDER BY avg_salary DESC
No parameters → params = {} → cache key never changes → single Redis entry serves all requests.

Top Earners — Parameterized LIMIT

The limit value is a bound parameter. Requesting top-5 and top-10 produce separate cache keys and are cached independently. A second call with the same limit is always a HIT.

Cache Control

Clearing the cache removes all dq:* keys from Redis using SCAN (safe for production — never blocks the event loop like KEYS *). After clearing, the next call to any query will be a MISS and re-populate the cache.

Cache Key Anatomy

Step 1
Serialise query name + params (sorted keys)
raw = f"{query_name}:{json.dumps(params, sort_keys=True)}"
# employees_filter:{"active_only": 0, "department": "Engineering",
#                   "hire_year_min": null, "sort_col": "salary_desc"}
Step 2
MD5 hash → short fixed-width key
digest = hashlib.md5(raw.encode()).hexdigest()   # 32 hex chars
key    = f"dq:{digest}"
# e.g. dq:a3f9b2c1d4e5f6a7b8c9d0e1f2a3b4c5
Why MD5?

MD5 is not used for security here — it's used purely for its fixed 32-char output. Any long SQL query + many parameters stays under Redis's 512 MB key limit and avoids key collisions in practice (the input space is small). SHA-256 is cryptographically stronger but MD5's 128-bit hash space is more than sufficient for cache key uniqueness.

sort_keys

json.dumps(params, sort_keys=True) guarantees that dict key ordering never affects the hash — Python dicts preserve insertion order (3.7+) and two equivalent parameter dicts assembled in different order must produce the same cache key.
