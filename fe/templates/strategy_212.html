Jitter + Circuit Breaker

Thundering Herd Problem Solutions

Unlike a stampede (one key, many requests), the thundering herd occurs when many different cache keys expire at the same moment — typically because they were all populated together at deployment or after a full cache flush. Every key expires simultaneously and a wave of requests floods the database all at once.

Solution A — TTL Jitter
import random

def jitter_ttl(base: int, pct=0.25) -> int:
    window = int(base * pct)
    return base + random.randint(-window, window)

# Each key gets a slightly different TTL
# → expiry spread over a ±25% window
# → DB load is naturally distributed
Solution B — Circuit Breaker
if cb.failures >= threshold:
    cb.state = OPEN      # stop all requests

if state == OPEN:
    if elapsed > recovery_s:
        state = HALF_OPEN  # let one probe through
    else:
        return default_data  # shield DB

if probe_succeeds:
    state = CLOSED  # back to normal

Key Insights

TTL Jitter
  • Add ±25% random offset when storing each key
  • Prevents synchronized mass expiry
  • Zero-code cost — just wrap your setex call
  • Doesn't help with a single hot key (→ use 2.11 for that)
Circuit Breaker
  • Shields DB during sustained failures
  • Three states: CLOSED → OPEN → HALF-OPEN → CLOSED
  • Prevents cascading failure avalanche
  • Recovery probe tests DB health before re-enabling
