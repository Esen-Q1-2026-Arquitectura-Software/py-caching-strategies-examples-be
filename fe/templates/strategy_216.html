Event-Driven Cache Invalidation

TTL expiry is a blunt instrument — data stays stale until the timer runs out. Event-driven invalidation fires exactly when data changes: a write publishes a message to a Redis Pub/Sub channel, a background listener on every app instance receives it and deletes the stale key immediately. Tag-based invalidation extends this: keys register under tag groups so an entire resource family can be purged with a single event.

Architecture — Data flow
PUT /product/3 → Update DB → PUBLISH cache:invalidate:ei216 → Listener receives event → DEL ei216:product:3 → Next GET: fresh DB fetch
Strategy 1 — Direct Key
Publish {"key": "ei216:product:3", "reason": "updated"} to cache:invalidate:ei216. Listener deletes exactly that key — surgical precision.
Strategy 2 — Tag Groups
Keys register under tag SETs (ei216:tags:products). Publish a tag name → listener scans the SET and deletes all associated keys in one batch.

Follow the golden path: 1) Fetch a product (MISS → cached), 2) Fetch again (HIT), 3) Update it (publishes invalidation), 4) Fetch once more (MISS again — cache was cleared by the listener).

How it works under the hood
1. Publisher

  "cache:invalidate:ei216",
  json.dumps({
    "key": key,
    "reason": "updated"
  })
)
2. Channel
Redis distributes the message to all subscribers on all connected instances — no polling, instant push delivery.
3. Subscriber
async for msg in pubsub.listen():
  key = data["key"]
  await redis.delete(key)

1% Knowledge — Event-Driven vs TTL

Why TTL alone is not enough
  • Data updated at T=0 stays stale until TTL fires at T=300 s
  • Under high read traffic, stale data is served thousands of times before expiry
  • Hard to tune: short TTL = frequent DB hits; long TTL = long staleness window
  • Flush-and-redeploy resets ALL TTLs simultaneously (cold start surge)
Event-driven advantages
  • Invalidation latency = Redis network latency (~1 ms), not TTL
  • Works across multiple app instances — every subscriber gets the event
  • Tag groups let one event clear a whole resource family (e.g., after a price import)
  • Combine: use TTL as safety net + events for instant precision
Phil Karlton's law — and the solution
"There are only two hard things in Computer Science: cache invalidation and naming things." Event-driven invalidation doesn't make the logic easier, but it makes the timing exact: the subscriber deletes the key within milliseconds of the write, without relying on the developer remembering to call redis.delete() in every code path that touches that data.
